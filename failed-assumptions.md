# Coffeebara 설계 회고

## 1. 처음 가정

- Kakao Local API + 추가 처리 + LLM이면 개인 취향 기반 카페 추천까지 자연스럽게 갈 수 있을 것이라 봤다.
- 선호 카페와 그 이유를 모으면 유저 취향 프로필을 만들 수 있다고 봤다.

## 2. 실제로 확인된 것

- 유저 취향(reason)은 구조화 가능하다.
- 하지만 후보 카페 쪽 데이터가 부족해서 추천/재정렬의 근거가 약하다.
- Kakao Local API는 장소 탐색에는 충분하지만, 카페의 성격을 비교할 정도의 풍부한 데이터를 주지 않는다.

## 3. 깨진 가정

- 유저 데이터만 있으면 추천까지 이어질 것이라는 가정
- Kakao 장소 데이터만으로 item-side 비교가 가능할 것이라는 가정
- 크롤링/스크래핑 + LLM이면 이 문제가 깔끔히 해결될 것이라는 가정

## 4. 왜 안 됐는가

- 추천은 user-side와 item-side가 둘 다 있어야 하는데, item-side 데이터가 빈약했다.
- LLM은 없는 사실을 안정적으로 만들어내지 못한다.
- 크롤링은 보강 수단이지 만능 해결책이 아니었다.

## 5. 그래서 바꾼 판단

- `추천 서비스`보다 `개인용 카페 탐색/기록 도구` 쪽이 현재 데이터 현실에 더 맞다.
- 선호 카페 저장, reason 기록, 태그/메모, 회고 기능은 그대로 의미가 있다.
- 추천은 보류하거나, 훨씬 약한 형태로 재정의해야 한다.

## 6. 남은 자산

- Kakao 기반 검색
- 지도 탐색
- 카페 저장
- 로그인 구조
- DB 적재 흐름

## 7. 얻은 교훈

- LLM을 붙인다고 item-side 문제가 자동으로 해결되지는 않는다.
- 추천 이전에 먼저 확인해야 할 것은 `비교 가능한 데이터가 양쪽에 있는가`이다.
- 프로젝트가 흔들릴 때는 코드보다 제품 정의를 먼저 다시 봐야 한다.
