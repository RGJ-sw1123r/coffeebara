# Coffeebara 제품 방향 노트

> English version: [product-direction-notes.md](./product-direction-notes.md)

## 1. 출발점

Coffeebara는 단순한 문제의식에서 시작되었습니다.  
커피를 탐독하고 음미하는 과정에서, 내가 어떤 종류의 카페를 선호하는지 스스로 파악하고 싶었고, 처음 가보는 국내 지역에서도 큰 수고 없이 내가 원하는 종류의 카페를 찾고 싶었습니다.

이것이 초기 제품 방향의 출발점이었습니다.

당시의 가정은 장소 검색, 저장한 카페, 취향 신호, 그리고 이후의 LLM 활용이 결합되면 개인화된 카페 추천 서비스로 발전할 수 있다는 것이었습니다.

우리는 내부적으로 이 초기 단계를 **v1**이라고 부릅니다.

## 2. 방향이 바뀐 이유

방향이 바뀐 이유는 추천이라는 아이디어가 상상하기는 쉬웠지만, 실제로 설득력 있게 뒷받침하기는 어려웠기 때문입니다.

주된 문제는 다음과 같았습니다.

- Kakao 장소 데이터는 검색과 참조에는 유용했지만, 추천 품질을 안정적으로 뒷받침하기에는 너무 얕았습니다.
- 사용자 취향 입력만으로는 추천 결과를 충분히 납득 가능한 수준으로 만들기 어려웠습니다.
- 더 강한 추천 품질을 만들려면 추가적인 아이템 측 데이터가 필요했지만, 현재 수집 구조에서는 자연스럽게 확보되지 않았습니다.
- 크롤링, 스크래핑, LLM 기반 보강은 나중에 도움이 될 수는 있어도, 그것만으로 제품의 중심을 정당화할 수는 없었습니다.

즉, 추천 방향은 데이터 측면에서 너무 약하고, 비용이 크며, 안정성이 낮았습니다.

## 3. v1에서 남아 있는 것

이전 작업이 헛수고였던 것은 아닙니다.

그 결과물은 현재도 저장소 안에 탐색 중심 구현과 아카이브 자료의 형태로 남아 있으며, 여전히 유효한 기술 자산입니다.  
일부 초기 작업물은 `docs/archive/` 아래에도 보존되어 있습니다.

그 단계는 더 이상 현재 제품의 기준선은 아니지만, 기술적 결과물은 필요할 때 재사용 가능한 상태로 남아 있습니다.

지금도 이어지는 자산은 다음과 같습니다.

- Kakao 로그인
- 장소 검색과 장소 저장 흐름
- `kakao_place_id` 기반 장소 식별 구조
- Spring Boot + MyBatis + MariaDB 애플리케이션 구조
- 기존 프론트엔드 상호작용 패턴
- 구현 메모와 AI 협업 기록

## 4. 현재 제품의 중심

현재 제품의 중심은 공개 추천이 아닙니다.

Coffeebara는 이제 다음과 같은 방향에서 더 타당합니다.

- 개인 커피 아카이브
- 브루잉 기록 시스템
- 장소 맥락을 가진 커피 로그
- 회상, 회고, 누적된 기록을 중심으로 한 제품

이 방향이 더 안정적인 이유는, 사용자가 실제로 직접 제공하고, 다시 돌아보며, 의미를 느낄 수 있는 데이터에 기반하기 때문입니다.

## 5. Kakao 자산의 현재 의미

Kakao 연동은 여전히 중요하지만, 그 역할은 달라졌습니다.

### Kakao 로그인

현재 역할은 다음과 같습니다.

- 사용자 식별
- 기록 소유권
- 기기 간 연속성 유지

### Kakao 장소 데이터

현재 역할은 다음과 같습니다.

- 원두를 구매한 장소 선택
- 커피를 마신 장소 선택
- 개인 기록에 장소 맥락 부여

이제 Kakao 자산은 추천 서비스의 중심이 아닙니다.

## 6. 현재 활성 범위

현재 제품에서 활성 범위는 다음처럼 이해해야 합니다.

- 회원 저장 카페
- 브라우저 로컬 상태 기반의 게스트 저장 카페
- 회원의 카페 연동 텍스트 기록
- 장소 맥락을 포함한 아카이브 흐름
- 아카이브 중심의 계정/프로필 구조

현재 저장 구조는 다음과 같습니다.

- `cafe`: 공용 장소 캐시/마스터 데이터
- `user_saved_cafe`: 회원 소유 저장 카페 상태
- 게스트 저장 카페: 임시 브라우저 상태
- `cafe_record`: 회원 소유 카페 연동 기록의 부모 레이어
- `cafe_note`: 현재 구현된 텍스트 기록 payload 레이어

앞으로의 원두 기록과 브루잉 기록도 같은 기록 중심 구조를 확장하는 방향이 되어야 합니다.

## 7. 검색, 지도, 게스트 모드

검색 및 지도 관련 코드는 여전히 저장소 안에 남아 있지만, 현재 제품의 기준선은 **지도 중심 탐색**이 아닙니다.  
현재의 기본 기준선은 **개인 아카이브 흐름**입니다.

검색과 장소 조회는 다음 정도의 범위를 유지하는 것이 맞습니다.

- 사용자가 실제로 검색하거나 조회한 것만 캐시하기
- 필요할 때만 상세 정보를 갱신하기
- 장소 수집 자체를 목적으로 과도하게 확장하지 않기

게스트 모드도 임시방편이 아니라 의도된 역할로 봐야 합니다.

게스트 모드의 역할은 다음과 같습니다.

- 즉시 계정 생성을 요구하지 않고 검색과 저장을 체험하게 하기
- 가벼운 브라우저 기반 경험 유지하기
- 정식 회원 기록 이전에 아카이브 방향을 미리 보여주기

다만 게스트 모드에는 분명한 한계가 있습니다.

- 게스트 데이터는 계정 기반의 영속 소유가 아닙니다.
- 게스트는 완전한 아카이브 소유가 아니라 미리보기/데모 경험을 위한 상태입니다.

## 8. AI의 위치

AI는 여전히 제품 안에서 의미가 있지만, 기반이 되어서는 안 됩니다.

의미 있는 AI 활용은 나중 단계의 회고 보조입니다. 예를 들면 다음과 같습니다.

- 반복적으로 나타나는 취향 경향 요약
- 만족도가 높았던 브루잉 조건 정리
- 사용자의 아카이브를 읽기 쉬운 형태로 요약

반대로 AI가 해서는 안 되는 일은 다음과 같습니다.

- 시음 노트를 대신 지어내는 것
- 기록 행위 자체를 대체하는 것
- 제품을 다시 AI 우선 추천 서비스로 되돌리는 것

아카이브는 AI 없이도 스스로 성립해야 합니다.

## 9. 다음 제품 슬라이스

다음으로 의미 있는 제품 단위는 다음과 같습니다.

- 게스트 샘플/데모 기록 페이지
- 원두 기록 생성
- 브루잉 기록 생성
- 구매 장소 및 음용 장소 선택
- 아카이브 상세 화면
- 시음 노트 편집
- 만족도 입력
- 필터링 및 정렬

이 항목들은 현실적으로 구현 가능한 크기이면서도, 제품을 실제로 개인적으로 유용하게 만드는 데 충분한 의미를 가집니다.

## 10. 최종 정리

Coffeebara는 같은 저장소 안에서 계속 이어져야 합니다.

내부적으로 **v1**이라고 부르는 초기 단계는 현재 제품의 기준선이 아니라, 보존된 구현 마일스톤으로 다뤄져야 합니다.  
현재 단계는 개인 브루잉 기록과 커피 아카이브 시스템에 집중해야 합니다.

목표는 과거 작업을 지우는 것이 아닙니다.  
목표는 기존 코드베이스가 더 강하고 더 오래 가는 제품 중심을 섬기도록 만드는 것입니다.
